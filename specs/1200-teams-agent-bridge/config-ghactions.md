# GitHub Actions Configuration — Spec 1200

Configuration required on the GitHub side so the `agent-react.yml`
workflow can authenticate its callback to the bridge service. This covers
secrets, workflow inputs, and the "Deliver callback" step.

For the Microsoft Teams / Azure side, see
[config-msteams.md](config-msteams.md).

## 1. Repository Secrets

The following secret must be configured in the repository (or
organization) that hosts `agent-react.yml`:

| Secret | Purpose | How to generate |
|---|---|---|
| `SERVICE_SECRET` | Shared HMAC-SHA256 key for inter-service authentication. The bridge verifies callback requests using this secret via `HmacAuth` from librpc. | Run `just env-setup` (which calls `scripts/env-setup.js`) — it generates a cryptographically random hex string of sufficient length (≥32 chars). Use the same value in the bridge's local `.env` and in the GitHub Actions secret. |

Set it via the GitHub UI (Settings > Secrets and variables > Actions >
New repository secret) or via the CLI:

```sh
gh secret set SERVICE_SECRET --body "$(node -e "console.log(require('crypto').randomBytes(32).toString('hex'))")"
```

The same `SERVICE_SECRET` value must be available to:

- **The bridge service** — via `process.env.SERVICE_SECRET` (loaded from
  `.env` by libconfig or exported in the shell).
- **GitHub Actions** — via the repository secret, passed to the "Deliver
  callback" step as an env var.

If the values differ, the bridge will reject every callback with a 401.

## 2. Workflow Dispatch Inputs

`agent-react.yml` accepts three `workflow_dispatch` inputs. The bridge
sends all three when dispatching a facilitate session:

| Input | Required | Description |
|---|---|---|
| `prompt` | yes | The user's message (with conversation history prepended by the bridge). |
| `callback_url` | no | Public URL for the bridge's callback endpoint, e.g. `https://<tunnel>/api/callback/<token>`. Omitting it skips the callback step — existing manual dispatches are unaffected. |
| `correlation_id` | no | UUID generated by the bridge per dispatch. Returned in the callback payload so the bridge can match the response to the originating conversation. |

The bridge constructs the `callback_url` as
`{callback_base_url}/api/callback/{random-uuid}` where the UUID is a
one-time routing token. The HMAC `Authorization` header — not the URL
token — is the authentication mechanism.

## 3. The "Deliver Callback" Step

After the facilitate session completes, a post-step in `agent-react.yml`
reads the NDJSON trace, extracts the orchestrator's verdict and summary,
and POSTs them to the `callback_url`.

The step runs when `callback_url` is non-empty and fires on both success
and failure of the facilitate session (`always()` condition) so the
caller is never left waiting.

### Authentication

The callback POST must include an `Authorization: Bearer <token>` header
where the token is generated by `HmacAuth` from `@forwardimpact/librpc`
using `SERVICE_SECRET`. The bridge creates its own `HmacAuth` instance
from the same secret and verifies the header before processing the
payload. Tokens are time-limited (60 s default lifetime in `HmacAuth`).

### Current state (requires update)

The `fit-eval callback` command
(`libraries/libeval/src/commands/callback.js`) and the `curl` fallback
in the workflow step do **not** yet include the `Authorization` header.
To complete the integration:

1. **`fit-eval callback`** — accept a `--service-secret` option (or read
   `SERVICE_SECRET` from the environment). Import `HmacAuth` from
   `@forwardimpact/librpc`, generate a token with
   `new HmacAuth(secret).generateToken("agent-react")`, and include it
   as `Authorization: Bearer <token>` in the POST headers.

2. **`curl` fallback** — generate the HMAC token via a Node one-liner in
   the shell step and pass it as a `-H 'Authorization: Bearer ...'`
   header.

3. **Workflow step env block** — expose the secret to the step:

   ```yaml
   - name: Deliver callback
     if: github.event_name == 'workflow_dispatch' && always() && inputs.callback_url != ''
     env:
       SERVICE_SECRET: ${{ secrets.SERVICE_SECRET }}
       CALLBACK_URL: ${{ inputs.callback_url }}
       CORRELATION_ID: ${{ inputs.correlation_id }}
       RUN_URL: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}
       TRACE_FILE: ${{ steps.assess.outputs.trace-file }}
     run: |
       # ... (existing logic, now with SERVICE_SECRET available)
   ```

Until these changes land, the bridge will reject callbacks with 401.

## 4. Other Existing Secrets

These secrets are already configured for `agent-react.yml` and are not
specific to the Teams bridge, but the bridge depends on them indirectly:

| Secret / Variable | Used by | Purpose |
|---|---|---|
| `GITHUB_TOKEN` (automatic) | The workflow itself | Permissions for checkout, issue/PR operations. The bridge uses a separate `GH_TOKEN` PAT for `workflow_dispatch` (the automatic token cannot trigger workflows). |
| `ANTHROPIC_API_KEY` | `fit-eval` step | Powers the Claude-based facilitate session. Not used by the bridge or callback step. |

## 5. Verification

After configuring `SERVICE_SECRET`:

1. Confirm the secret is set:

   ```sh
   gh secret list | grep SERVICE_SECRET
   ```

2. Trigger a test dispatch with a callback URL pointing to a request
   inspector (e.g. [webhook.site](https://webhook.site)):

   ```sh
   gh workflow run agent-react.yml \
     -f prompt="test" \
     -f callback_url="https://webhook.site/<your-uuid>" \
     -f correlation_id="test-123"
   ```

   After the run completes, check whether the callback arrived with an
   `Authorization: Bearer ...` header. If the header is absent, the
   `fit-eval callback` command has not yet been updated (see § 3).

3. Run the full round-trip: start the bridge locally with the same
   `SERVICE_SECRET`, send a message in Teams, and verify the facilitator
   response arrives in the same thread.
